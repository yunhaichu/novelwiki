# Outline Review Gate

Use this after generating or revising a project outline and before creating volume plans or chapter beats.

## Decision

Return one of:

- PASS
- REVISE
- FAIL

## Required Checks

### 1. Type Contract Alignment

If a type contract exists, the outline must serve the declared primary reader promise.

The outline must state:

- primary reader promise;
- what chapter 1 must show;
- what chapter 1 must delay;
- what chapter 3 must prove;
- type drift risks;
- false promise risks.

Mark REVISE if the outline has a genre label but no reader-facing promise.

Mark REVISE if the opening and ending appear to pay different primary promises without a controlled cross-genre plan.

### 2. Core Conflict

The outline must have:

- protagonist goal
- opposing force
- failure cost
- irreversible pressure

Mark REVISE if the core conflict is only a setting description.

### 3. Reader Entry Feasibility

The outline must support a chapter 1 that a first-time reader can enter without wiki knowledge.

A future reader entry gate must be able to make these visible:

```text
protagonist
immediate trouble
present desire
stake
primary story promise
reason to continue
```

Mark REVISE if the opening depends on world explanation, backstory, travel, status description, or ordinary daily life before imbalance.

### 4. Opening Hook

The first strong hook must appear in chapter 1 or 2.

Mark REVISE if the hook is only strange, atmospheric, symbolic, or vague and does not create pressure, cost, choice, proof path, threat, or relationship movement.

### 5. First Three Chapters

The first three chapters must contain at least three visible state changes.

A state change can be:

- status change
- resource change
- relationship change
- information change
- risk change
- method change
- belief change
- choice that closes one path and opens another

Mark REVISE if the first three chapters are mainly setup, setting tour, local tasks, or mystery accumulation without visible reader payoff.

### 6. Five Turning Points

The outline must include:

- imbalance
- first tool or method
- relationship reconfiguration
- truth turn
- terminal choice

Mark REVISE if these are only summaries and do not change goal, risk, relation, method, or rule.

### 7. Relationship Matrix

Each core relationship must include:

- surface relation
- hidden tension
- expected change

Mark REVISE if side characters only support, explain, mock, praise, or accompany the protagonist.

### 8. Three-Layer Suspense

The outline must have:

- immediate problem
- midterm mystery or pressure
- longterm truth or endpoint

Mark REVISE if all suspense is delayed to the ending.

Mark REVISE if the immediate layer is not understandable without later explanation.

### 9. Main Genre Promise

The story engine must connect to the declared main genre promise.

The outline must define what counts as progress for this specific project.

Examples:

```text
cultivation/progression: power, resources, rule understanding, survival in sect/system
urban/workplace: money, status, relationship, career, reputation, legal/social pressure
science fiction: anomaly proof, rule discovery, technical constraint, human cost
mystery: clue, contradiction, suspect relation, proof chain
romance: relationship movement, emotional risk, intimacy, commitment, repair
farming/management: resource chain, production, land, community, defense, market access
officialdom/political: position, alliance, procedure, policy risk, reputation
thriller: threat escalation, narrowed options, survival move, enemy adaptation
```

Mark REVISE if the outline pays only a side promise while ignoring the declared main promise.

### 10. Consequence And Escalation Feasibility

The outline must support chapters where events create consequences and conflict changes over time.

Mark REVISE if the story appears to proceed through a chain of tasks, discoveries, or episodes without:

- stronger opposition;
- higher cost;
- reduced retreat path;
- changed method;
- relationship or status consequence;
- forced next action.

### 11. Drift Risk

The outline must name its most likely drift and how to prevent it.

Common drifts:

- task-of-the-day structure
- setting explanation without story movement
- reader-entry failure
- repeated humiliation/payoff loop
- protagonist solving too cleanly
- side characters becoming fixed functions
- low-anchor terms becoming fake worldbuilding
- side genre overtaking the main genre promise without compensation
- hook becoming weird but not compelling

## Output Format

```text
Main reason:

```
